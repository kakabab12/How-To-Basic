# 5챕터 — 1층 capture — 카메라는 왜 별도 스레드인가

> **광명테크 제스처 인식** · 5/15챕터 · [📚 목차](README.md)

---

`src/capture/camera_stream.py`

**문제**: 카메라는 초당 30장을 뱉는데 추론이 초당 10장밖에 못 한다. 그냥 순서대로 읽으면
OpenCV 내부 버퍼에 프레임이 쌓여서, 내가 손을 든 지 3초 뒤에 그 장면이 처리된다.

**해결**: 캡처 전용 스레드가 계속 읽어서 **최신 한 장만 덮어쓴다.**

```python
def _capture_loop(self):
    while self.is_running:
        ret, frame = self._cap.read()
        if not ret:
            time.sleep(0.1)
            continue
        with self._frame_lock:
            self._frame = frame     # ← 쌓지 않고 덮어쓴다
        self.fps_meter.update()

def capture_frame(self):
    deadline_sec = time.monotonic() + FIRST_FRAME_TIMEOUT_SEC   # 5초
    while True:
        with self._frame_lock:
            if self._frame is not None:
                return self._frame.copy()   # ← 복사본을 준다
        if time.monotonic() > deadline_sec:
            raise RuntimeError("카메라에서 프레임을 받지 못했습니다 (연결/장치 번호 확인)")
        time.sleep(0.01)
```

읽어낼 포인트 3가지:

- **`Lock`으로 감싼 이유** — 캡처 스레드가 쓰는 도중에 메인 스레드가 읽으면 찢어진 프레임이 나온다.
- **`.copy()`를 주는 이유** — 원본을 그대로 주면 처리하는 동안 캡처 스레드가 덮어써 버린다.
- **5초 타임아웃** — 첫 프레임은 카메라 워밍업 때문에 늦다. 무한 대기 대신 명확한 에러를 낸다.

**플랫폼별 분기**가 `init_camera()`에 있는데, 이건 실제로 겪은 문제의 기록이다:
