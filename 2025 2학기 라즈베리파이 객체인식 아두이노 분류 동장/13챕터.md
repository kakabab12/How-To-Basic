```python
tx, ty, tz = best_candidate_in_frame['real_coords']
px, py = best_candidate_in_frame['center']       # 화면 픽셀 좌표
t_class = most_common_class

# 🔥 LOCK 좌표를 정확히 무게 중심(px, py)으로 설정
locked_target = { "x": px, "y": py, "z": tz, "class": t_class }
print(f"⚡ [타겟 확정] {t_class} (중심: {px},{py})")

# 번호 매핑 (1~6번)
sort_type = 0
if   t_class == "small_normal_box":    sort_type = 1
elif t_class == "medium_normal_box":   sort_type = 2
elif t_class == "big_normal_box":      sort_type = 3
elif t_class == "small_abnormal_box":  sort_type = 4
elif t_class == "medium_abnormal_box": sort_type = 5
elif t_class == "big_abnormal_box":    sort_type = 6

if ser and ser.is_open:
    cmd = f"SORT:{sort_type}\n"
    ser.write(cmd.encode())
    print(f"📤 아두이노 전송: {cmd.strip()}")

# 타겟 유지 시간 3.0초
threading.Timer(3.0, release_target).start()
```

#### 통신 규격 — 왜 이렇게 단순한가

**`SORT:3\n`** — 문자열 한 줄. 아두이노는 8비트 MCU라 JSON 파서를 돌릴 여유가 없다.
`Serial.readStringUntil('\n')`으로 한 줄 읽고 숫자만 뽑으면 끝이다.

**1~6 매핑에 순서가 있다**: 정상(1~3)이 먼저, 불량(4~6)이 나중. 각 그룹 안에서 small→big.
아두이노 쪽에서 `if (n <= 3) 정상라인; else 불량라인;` 같이 쓸 수 있는 배치다.

#### `app_base.py`와의 차이 — 설계 변경의 기록

`app_base.py`는 **다른 규격**을 쓰려 했다:

```
app_base.py : MOVE:x,y,z     ← 3D 좌표를 보내 로봇이 그 지점으로 이동
sucess.py   : SORT:1~6       ← 분류 번호만 보내 로봇이 정해진 칸으로
```

**왜 바꿨나**: `MOVE:x,y,z`는 카메라 좌표 → 로봇 좌표 캘리브레이션이 정확해야 한다.
`SORT:n`은 **로봇이 미리 정해진 6개 위치만 알면 된다**. 훨씬 단순하고 오차에 강하다.
`real_x`, `real_y` 계산이 코드에 **남아 있지만 실제로는 로그에만 쓰인다**(11단계) —
그 설계의 잔재다.

#### 3초 락

```python
threading.Timer(3.0, release_target).start()

def release_target():
    global locked_target
    with data_lock:
        locked_target = None
        class_vote_buffer.clear()
        print("\n🔓 [시스템] 타겟 고정 해제 -> 재탐색 시작\n")
```

**타겟을 확정하면 3초간 얼어붙는다.** 그동안:
- 추론 루프가 통째로 쉰다 (8단계)
- 로봇이 박스를 집어 간다
- 3초 후 `Timer`가 락을 풀고 **버퍼를 비운다**(`clear()`) — 이전 박스의 표가 다음 판정에
  섞이면 안 되니 여기서는 통째로 비우는 게 맞다

`threading.Timer`는 **비동기 타이머**라 추론 루프를 블로킹하지 않는다.
