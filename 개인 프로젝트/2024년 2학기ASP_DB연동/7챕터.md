# 7챕터 — `delete.asp` — Access 방언

> **ASP + Access DB 연동 (2024)** · 7/10챕터 · [📚 목차](README.md) · [개인 프로젝트 전체](../README.md)

---

**15줄, 가장 짧은 파일**이다:

```asp
<%
X1=Request.Form("txtcid")

DSNless="DRIVER={Microsoft Access Driver (*.mdb)}; "
DSNless=DSNless & "DBQ=" & server.mappath("nwind.mdb")

Set Conn = Server.CreateObject("ADODB.Connection")
Conn.Open DSNless

sql = "DELETE * FROM tblCategories WHERE CategoryID = '"& X1 &"';"
Conn.execute(sql)
Conn.Close

Response.Redirect "delete2.asp"
%>
```

#### `DELETE *` 가 눈에 띈다

**표준 SQL은 `DELETE FROM`이다:**

```sql
DELETE FROM tblCategories WHERE ...      -- 표준 (SQL Server, MySQL, PostgreSQL)
DELETE * FROM tblCategories WHERE ...    -- Access 방언
```

**`*`는 Access(Jet SQL)에서만 허용**된다. 다른 DB에 옮기면 **문법 오류**가 난다.

> **"DB마다 SQL이 조금씩 다르다"**를 보여주는 실물 예시다. 표준 SQL이 있어도
> 각 DB가 자기 방언을 갖는다. Access의 `*`, MySQL의 백틱, SQL Server의 `TOP` 등.

#### 셋 다 없다

| 없는 것 | 결과 |
|---|---|
| **`Set Conn = nothing`** | COM 객체를 명시적으로 안 놓는다 (3단계) |
| **삭제 확인** | 버튼 누르면 바로 지운다 |
| **결과 확인** | 몇 건 지워졌는지, 실패했는지 모른다 |

**`Conn.execute(sql)`은 영향받은 행 수를 반환할 수 있는데** 안 받는다:

```asp
Conn.Execute sql, recordsAffected
' recordsAffected 에 삭제된 행 수가 담긴다
```

**존재하지 않는 ID를 넣어도 조용히 성공**한다.

> **고클린 배치 버전**([7단계](../project고클린_초기.ver/README.md))이 게임 모드에
> `set /p confirm=` Y/N 확인을 넣은 것과 대조된다. **파괴적 작업에 확인을 붙이는 습관**은
> 개인 프로젝트에서는 있었는데 학교 실습 코드에는 없다 — 실습의 목적이 다르니 당연하다.

#### `WHERE`가 있는 건 다행이다

```sql
DELETE * FROM tblCategories WHERE CategoryID = '...'
```

**`WHERE`를 빠뜨리면 테이블 전체가 지워진다.** 흔한 사고인데 여기는 있다.
다만 `X1`이 비어 있으면 `WHERE CategoryID = ''`가 되어 **아무것도 안 지운다**(안전).

**단, 10단계의 SQL 인젝션으로는 이 `WHERE`를 우회할 수 있다.**
