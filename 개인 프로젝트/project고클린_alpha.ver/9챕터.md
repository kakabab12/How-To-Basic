# 9챕터 — ★ `safe_decode` — 한글 Windows 인코딩 지옥

> **고클린 alpha.ver** · 9/15챕터 · [📚 목차](README.md) · [개인 프로젝트 전체](../README.md)

---

```python
def safe_decode(self, byte_data):
    try: return byte_data.decode('cp949')
    except: pass
    try: return byte_data.decode('utf-8')
    except: pass
    return byte_data.decode('cp949', errors='replace')
```

**5줄짜리 함수인데, 한국어 Windows에서 `subprocess`를 쓰면 반드시 만나는 문제**를 푼다.

#### 문제

| 무엇 | 인코딩 |
|---|---|
| 한국어 Windows CMD 출력 | **CP949** (= 확장 완성형, MS949) |
| Python `bytes.decode()` 기본값 | **UTF-8** |
| 일부 최신 도구(`winget` 등) 출력 | UTF-8 |

```python
output = subprocess.check_output("ipconfig", shell=True)
output.decode()          # UnicodeDecodeError!  ← 프로그램 죽음
```

`ipconfig`가 **"이더넷 어댑터"**를 CP949로 출력하는데 UTF-8로 읽으려다 터진다.

#### 3단 폴백

```
1. cp949 시도            ← 한국어 Windows 명령 대부분 (ipconfig, wmic, ping...)
   ↓ 실패
2. utf-8 시도            ← 최신 도구 (winget 등)
   ↓ 실패
3. cp949 + errors='replace'   ← 마지막 수단: 깨진 바이트는 '' 로 대체
                                 절대 예외를 던지지 않는다
```

**3번이 이 함수의 핵심**이다. `errors='replace'`는 디코딩 실패한 바이트를 `` 로 바꾼다.

> **판단**: 글자 몇 개가 `` 로 깨지는 것 ≪ 프로그램이 죽는 것.
> 사용자에게는 **깨진 글자가 섞인 로그**가 **크래시**보다 백 배 낫다.

#### 순서가 중요하다

**CP949를 먼저 시도**한다. 반대로 하면 안 되는 이유:

UTF-8 디코딩은 **아무 바이트나 받아들이지 않는다**(엄격한 규칙이 있어 CP949 한글은 대개
실패한다). 하지만 **CP949는 거의 모든 바이트를 어떻게든 해석**해 버린다 — UTF-8 문자열을
CP949로 읽으면 예외 없이 **깨진 한글(뷁어)**이 나온다.

```
"한글" (UTF-8 bytes) → cp949로 읽으면 → "ÇÑ±Û" (예외 없음!)
```

**즉 `cp949`를 나중에 두면 UTF-8 출력이 조용히 깨진다.**
지금 순서(`cp949` → `utf-8`)는 **한국어 Windows 명령이 압도적 다수**라는 판단에 맞다.

| | 초기.ver | alpha.ver |
|---|---|---|
| 접근 | **`chcp 65001`** — "CMD를 UTF-8로 바꾸자" | **`safe_decode`** — "뭐가 오든 받아내자" |
