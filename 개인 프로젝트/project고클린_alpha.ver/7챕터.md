# 7챕터 — ★ 멀티스레딩 — GUI가 얼어붙지 않게

> **고클린 alpha.ver** · 7/15챕터 · [📚 목차](README.md) · [개인 프로젝트 전체](../README.md)

---

**이 프로그램에서 가장 중요한 설계 결정**이다.

#### 문제 — "응답 없음"

```python
def start_clean(self):
    subprocess.check_output("defrag C: /O", shell=True)   # 3분 걸림
```

이렇게 쓰면 **3분 동안 창이 하얗게 변하고 제목에 "(응답 없음)"**이 붙는다.
Windows가 프로그램이 죽은 줄 알고 "프로그램 종료" 대화상자를 띄우기도 한다.

#### 왜 그런가 — 이벤트 루프

GUI는 **단일 스레드 이벤트 루프**로 돈다:

```
    ┌─────────────────────────────┐
    │  이벤트 대기                 │
    │      ↓                      │
    │  버튼 클릭 감지              │
    │      ↓                      │
    │  핸들러 실행  ← 여기서 3분 잡으면
    │      ↓            루프가 멈춘다
    │  화면 다시 그리기 ← 실행 안 됨
    │      ↓                      │
    └──────┘ (반복)               │
```

**화면을 다시 그리는 것도 이 루프의 일**이다. 핸들러가 루프를 붙잡고 있으면
창을 옮겨도, 다른 창에 가렸다 돌아와도 **다시 그려지지 않는다.** 그게 "하얀 창"의 정체다.

#### 답 — 무거운 일은 전부 스레드로

본인 설명:

> 명령 프롬프트(subprocess) 명령이 끝날 때까지 기다리면 GUI 프로그램이 "응답 없음" 상태가
> 되며 멈춥니다. 이를 방지하기 위해 모든 기능이
> `threading.Thread(target=task, daemon=True).start()` 형태로 분리되어 백그라운드에서
> 안전하게 돌아갑니다.

```python
def start_ping(self):
    self.textbox.delete("0.0", "end")                  # ① 즉시 UI 갱신
    self.set_working_state(True, "PING TEST...")       #    (이벤트 루프에서 실행)
    self.log(msg)

    def task():                                         # ② 무거운 일은 안쪽 함수에
        gms = get_ms("8.8.8.8")
        cms = get_ms("1.1.1.1")
        self.log_summary(...)
        self.set_working_state(False)                   # ④ 끝나면 UI 복구

    threading.Thread(target=task, daemon=True).start()  # ③ 백그라운드로 던지고 즉시 반환
```

**핸들러는 스레드를 시작하자마자 반환**한다 → 이벤트 루프가 살아난다 → 창이 계속 그려진다.

#### 패턴이 5개 기능 전부 동일하다

```
def start_XXX(self):
    self.set_working_state(True, "...")      # UI 갱신
    def task():                              # 클로저
        ...무거운 일...
        self.set_working_state(False)        # UI 복구
    threading.Thread(target=task, daemon=True).start()
```

**일관된 패턴은 그 자체가 문서다.** 한 개를 이해하면 나머지 넷은 볼 필요가 없다.

#### 세부

| 요소 | 이유 |
|---|---|
| **`daemon=True`** | 메인 창을 닫으면 스레드도 같이 죽는다. 없으면 `defrag`가 끝날 때까지 프로세스가 안 죽어 **작업관리자에 유령이 남는다** |
| **클로저 `def task()`** | `self`와 지역 변수를 그대로 캡처 → 인자 전달이 필요 없다 |
| **`.start()`** | `.run()`이 아니다. `.run()`은 **현재 스레드에서 그냥 실행**되어 아무 효과가 없다 (흔한 실수) |

> **⚠ 엄밀히 말하면 이 코드는 tkinter의 규칙을 어긴다.** tkinter 위젯은
> **메인 스레드에서만** 조작해야 하는데, `task()` 안에서 `self.log()`와
> `self.set_working_state()`를 부른다. 정석은 `self.after(0, lambda: ...)`로
> 메인 스레드에 예약하는 것이다.
>
> **실무에서 대부분 잘 돌아가지만**(tkinter가 관대한 편) 드물게 크래시할 수 있다.
> 이 규모의 개인 도구에서는 흔히 넘어가는 부분이다.
