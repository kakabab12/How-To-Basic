# 11챕터 — 다국어(i18n) 설계

> **고클린 alpha.ver** · 11/15챕터 · [📚 목차](README.md) · [개인 프로젝트 전체](../README.md)

```python
self.current_lang = "EN"       # ★ 기본값이 영어

# --- 언어 설정 데이터 ---
self.ui_locales = {
    "update":   {"EN": "  >  SYSTEM UPDATE", "KO": "  >  시스템 업데이트"},
    "clean":    {"EN": "  >  CLEAN & BOOST", "KO": "  >  최적화 및 부스트"},
    "game":     {"EN": "  >  GAME MODE",     "KO": "  >  게임 모드"},
    "ping":     {"EN": "  >  PING TEST",     "KO": "  >  핑 테스트"},
    "info":     {"EN": "  >  MY PC INFO",    "KO": "  >  내 PC 정보"},
    "ready":    {"EN": "SYSTEM READY",       "KO": "시스템 준비 완료"},
    "done":     {"EN": "[OK] JOB FINISHED",  "KO": "[완료] 작업 끝"},
    "lang_btn": {"EN": "한글", "KO": "ENG"},
}

# --- CMD 출력 메시지용 데이터 ---
self.cmd_locales = { ... }
```

#### 설계 포인트 4가지

**① 딕셔너리를 둘로 나눴다**

| 딕셔너리 | 대상 |
|---|---|
| `ui_locales` | 버튼 라벨, 상태 텍스트 |
| `cmd_locales` | 터미널에 찍히는 로그 |

**성격이 다르다.** UI 텍스트는 짧고 고정, 로그는 길고 값이 끼어든다. 나눈 게 맞다.

**② `{키: {언어: 문자열}}` 구조**

```python
"update": {"EN": "...", "KO": "..."}      # ← 키 중심 (현재)
# vs
"EN": {"update": "...", "clean": "..."}   # ← 언어 중심
```

**키 중심이 확장에 유리하다.** 일본어를 추가하려면 각 항목에 `"JA": "..."` 한 줄씩만
넣으면 된다. 언어 중심이면 **딕셔너리를 통째로 복사**해야 하고, 번역이 빠진 키를 놓치기 쉽다.

**③ `lang_btn`이 일부러 반대다**

```python
"lang_btn": {"EN": "한글", "KO": "ENG"}
```

**현재 영어면 버튼에 "한글"**이라고 쓴다. 누르면 한글이 되니까.
"현재 언어"가 아니라 **"누르면 될 언어"**를 표시한다 — 언어 전환 버튼의 표준 UX다.

**④ 기본값이 영어**

```python
self.current_lang = "EN"
```

한국 사람이 만든 도구인데 **영어로 시작**한다.
`.exe`로 배포할 것(15단계)과 **후원 기능**(14단계)을 넣은 걸 보면,
처음부터 **한국 밖 사용자**를 염두에 뒀다.

#### 실시간 전환

```python
def toggle_language(self):
    ...   # 모든 위젯의 텍스트를 다시 configure
```

**재시작 없이 즉시 바뀐다.** 6단계에서 **모든 위젯을 `self.`로 붙잡아 둔 덕분**이다.
`configure(text=...)`로 하나씩 갈아 끼운다.

> **한계**: `toggle_language`가 **위젯을 하나씩 수동으로** 갱신한다. 위젯이 늘면
> 여기도 같이 늘려야 하고, 빠뜨리면 그 위젯만 이전 언어로 남는다.
> 규모가 커지면 `{위젯: 키}` 매핑을 만들어 루프 도는 방식이 낫다.
