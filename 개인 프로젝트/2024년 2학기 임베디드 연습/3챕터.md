**보고서에서 추출한 전체 코드:**

```python
import time
import RPi.GPIO as GPIO
from picamera2 import Picamera2
import cv2

# GPIO 핀 번호 설정
LED_PIN = 17          # LED 핀 (GPIO 17 사용)

# GPIO 설정
GPIO.setmode(GPIO.BCM)
GPIO.setup(LED_PIN, GPIO.OUT)
GPIO.output(LED_PIN, GPIO.LOW)      # 시작 시 LED 꺼짐 상태

# Picamera2 초기화
picam2 = Picamera2()

def video_stream():
    try:
        # 카메라 시작
        picam2.start()
        print("카메라가 켜졌습니다.")
        GPIO.output(LED_PIN, GPIO.HIGH)     # 카메라가 켜지면 LED 켜기

        # 실시간 비디오 스트리밍
        while True:
            frame = picam2.capture_array()          # 카메라에서 프레임 캡처
            frame_resized = cv2.resize(frame, (640, 480))   # 영상 크기 조정
            cv2.imshow("Live Video", frame_resized)         # 영상 출력

            # 'q' 키를 눌러 종료
            if cv2.waitKey(1) & 0xFF == ord('q'):
                break

    except KeyboardInterrupt:
        print("프로그램이 중단되었습니다.")

    finally:
        # 카메라 중지 및 정리
        picam2.stop()
        print("카메라가 꺼졌습니다.")
        GPIO.output(LED_PIN, GPIO.LOW)      # 카메라가 꺼지면 LED 끄기
        cv2.destroyAllWindows()
        GPIO.cleanup()

if __name__ == "__main__":
    video_stream()
```

#### `finally` 블록이 이 코드의 핵심이다

```python
finally:
    picam2.stop()
    GPIO.output(LED_PIN, GPIO.LOW)
    cv2.destroyAllWindows()
    GPIO.cleanup()
```

**하드웨어 프로그래밍에서 `finally`는 선택이 아니라 필수**다.

**정리를 안 하면:**

| 안 한 것 | 결과 |
|---|---|
| `picam2.stop()` 누락 | **카메라가 잠긴 채 남는다** → 다음 실행 시 "카메라 사용 중" 에러 |
| `GPIO.output(LED_PIN, LOW)` 누락 | **LED가 계속 켜진 채** 방치 |
| `GPIO.cleanup()` 누락 | **핀 설정이 남아** 다음 프로그램이 경고를 낸다 |

**`Ctrl+C`(KeyboardInterrupt)로 끄든, `q`로 끄든, 예외로 죽든 — `finally`는 항상 실행된다.**

> **[광명테크 프로젝트](../../광명테크/README.md)의 `run_demo.py`가 정확히 같은 구조다:**
> ```python
> try:
>     while True:
>         annotated, _event = runner.step()
>         cv2.imshow(WINDOW_NAME, annotated)
>         if cv2.waitKey(1) & 0xFF == ord("q"):
>             break
> finally:
>     runner.stop()              # ← 카메라 반드시 놓기
>     cv2.destroyAllWindows()
> ```
> **2년 뒤 인턴십 코드와 뼈대가 같다.** `while True` + `waitKey(1) == ord('q')` + `finally` 정리.
> **이때 배운 패턴을 계속 쓰고 있다.**

#### `GPIO.setmode(GPIO.BCM)`

**라즈베리파이 핀 번호 체계가 두 가지**다:

| 모드 | 기준 |
|---|---|
| **`GPIO.BCM`** | **칩(Broadcom) 기준 GPIO 번호** ← 이 코드 |
| `GPIO.BOARD` | 물리적 핀 위치 (1~40번) |

**`LED_PIN = 17`은 "GPIO 17"**이지 17번째 핀이 아니다(물리적으로는 11번 핀).
**섞어 쓰면 엉뚱한 핀을 건드린다.** 그래서 `setmode`를 맨 위에 선언한다.

#### `capture_array()` → `cv2.resize()`

```python
frame = picam2.capture_array()                   # numpy 배열로 받는다
frame_resized = cv2.resize(frame, (640, 480))    # OpenCV로 크기 조정
```

**Picamera2는 numpy 배열을 직접 준다** — OpenCV와 바로 호환된다.
`cv2.VideoCapture`를 쓰지 않는 이유는 라즈베리파이 카메라 모듈이 전용 라이브러리를
요구하기 때문이다.

> **2025년 라즈베리파이 프로젝트의 `camera.py`가 같은 라이브러리를 쓴다:**
> ```python
> from picamera2 import Picamera2
> from libcamera import controls
> ...
> self.picam2.set_controls({"AwbMode": ..., "AfMode": controls.AfModeEnum.Continuous})
> ```
> **1년 뒤에는 오토포커스·화이트밸런스까지 제어한다.** 여기서는 `start()`만 한다.
> **같은 도구를 점점 깊이 쓰게 되는 과정**이 보인다.

#### `cv2.waitKey(1) & 0xFF == ord('q')`

**OpenCV의 관용구**다:

| 조각 | 이유 |
|---|---|
| `waitKey(1)` | **1ms 대기** — 이게 없으면 `imshow` 창이 안 그려진다 |
| `& 0xFF` | 하위 8비트만 — **일부 플랫폼에서 상위 비트에 쓰레기가 붙는다** |
| `ord('q')` | 문자 `'q'`의 아스키 코드 |

**`waitKey`가 없으면 창이 하얗게 뜬다.** OpenCV의 GUI 이벤트 처리가 `waitKey` 안에서
일어나기 때문이다 — [고클린의 이벤트 루프 문제](../project고클린_alpha.ver/README.md)와
같은 원리다.
