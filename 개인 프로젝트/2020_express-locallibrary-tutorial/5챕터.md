# 5챕터 — `app.js` — 앱 조립과 미들웨어 순서

> **Express 지역 도서관 (2020)** · 5/10챕터 · [📚 목차](README.md) · [개인 프로젝트 전체](../README.md)

---

```javascript
var createError = require('http-errors');
var express = require('express');
var path = require('path');
var cookieParser = require('cookie-parser');
var logger = require('morgan');

var indexRouter = require('./routes/index');
var usersRouter = require('./routes/users');

var app = express();

// (MongoDB 연결 — 6단계)

// view engine setup
app.set('views', path.join(__dirname, 'views'));
app.set('view engine', 'pug');

app.use(logger('dev'));                                    // ① 로깅
app.use(express.json());                                   // ② JSON 파싱
app.use(express.urlencoded({ extended: false }));          // ③ 폼 파싱
app.use(cookieParser());                                   // ④ 쿠키
app.use(express.static(path.join(__dirname, 'public')));   // ⑤ 정적 파일

app.use('/', indexRouter);                                 // ⑥ 라우터
app.use('/users', usersRouter);

// catch 404 and forward to error handler
app.use(function(req, res, next) {                         // ⑦ 404
  next(createError(404));
});

// error handler
app.use(function(err, req, res, next) {                    // ⑧ 에러 (인자 4개!)
  res.locals.message = err.message;
  res.locals.error = req.app.get('env') === 'development' ? err : {};
  res.status(err.status || 500);
  res.render('error');
});

module.exports = app;
```

#### `app.use()` 순서가 곧 실행 순서다

**Express의 미들웨어는 "파이프라인"**이다. 요청이 위에서 아래로 흐른다:

```
요청 들어옴
   ↓
① morgan     — 콘솔에 "GET / 200 12ms" 찍음
   ↓
② express.json() — Content-Type: application/json 이면 req.body 채움
   ↓
③ urlencoded  — 폼 전송이면 req.body 채움
   ↓
④ cookieParser — req.cookies 채움
   ↓
⑤ static      — /stylesheets/style.css 같은 게 있으면 여기서 응답하고 끝!
   ↓  (없으면 계속)
⑥ 라우터      — '/' 면 indexRouter로
   ↓  (매칭 안 되면 계속)
⑦ 404 핸들러  — 아무도 안 받았다 = 없는 페이지
   ↓
⑧ 에러 핸들러 — next(err) 가 불리면 여기로 점프
```

**순서를 바꾸면 깨진다:**

| 잘못된 순서 | 결과 |
|---|---|
| 라우터를 `static` 앞에 | CSS 요청이 라우터로 가서 404 |
| `urlencoded`를 라우터 뒤에 | `req.body`가 `undefined` |
| 404를 라우터 앞에 | **모든 요청이 404** |
| 에러 핸들러를 위에 | 에러를 못 잡음 |

**⑦ 404가 "맨 마지막 라우트"인 이유**: Express는 위에서부터 매칭을 시도하는데,
**아무 라우터도 안 받으면** 여기까지 흘러온다. 즉 **"여기 도달했다 = 없는 주소다"**.

#### ⑧ 에러 핸들러는 인자가 4개다

```javascript
app.use(function(err, req, res, next) {   // ← err 가 첫 번째!
```

**Express는 함수의 인자 개수로 에러 핸들러를 구분한다.** 4개면 에러 핸들러,
3개면 일반 미들웨어다. **`err`를 빼면 그냥 미들웨어가 되어 에러를 못 잡는다** —
Express의 유명한 함정이다.

```javascript
res.locals.error = req.app.get('env') === 'development' ? err : {};
```

**개발 모드에서만 에러 상세(스택 트레이스)를 보여준다.** 프로덕션에서는 빈 객체 —
**스택 트레이스는 공격자에게 서버 구조를 알려주므로** 숨기는 게 맞다.
`express-generator`의 기본 코드지만 좋은 관습이다.
