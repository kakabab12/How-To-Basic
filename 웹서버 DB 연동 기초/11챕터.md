# 11챕터 — 실습 ② 스키마와 SQL 기초

> **웹 서버 + DB 연동 기초** · 11/15챕터 · [📚 목차](README.md)

---

#### `schema.sql`을 왜 따로 빼나

```sql
DROP TABLE IF EXISTS guestbook;

CREATE TABLE guestbook (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    name      TEXT    NOT NULL,
    message   TEXT    NOT NULL,
    created_at TEXT   NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_guestbook_created_at ON guestbook (created_at DESC);
```

| 이유 | 설명 |
|---|---|
| **한눈에 보인다** | 코드 안에 흩어져 있으면 구조를 알 수 없다 |
| **SQL만 읽어도 안다** | 파이썬을 몰라도 DB 구조를 파악 가능 |
| **이식이 쉽다** | MySQL로 옮길 때 **이 파일만** 고치면 된다 (7단계의 방언) |

#### 한 줄씩 뜯어보기

**`INTEGER PRIMARY KEY AUTOINCREMENT`**

```
INTEGER PRIMARY KEY  → SQLite 에선 이것만으로 자동증가가 된다 (rowid 별칭)
AUTOINCREMENT        → 재사용 안 함을 보장 (삭제된 id 를 다시 안 씀)
```

**MySQL이면 `INT AUTO_INCREMENT`, PostgreSQL이면 `SERIAL`**이다.

**`NOT NULL` — DB가 강제하는 검증**

```python
# app.py 에서도 검증하는데 왜 DB 에서도?
if not name or not message:
    return redirect(url_for("index"))
```

**애플리케이션 검증은 우회될 수 있다.** 다른 프로그램이 같은 DB에 붙거나,
누가 `sqlite3` CLI로 직접 넣으면? **DB 제약은 마지막 방어선**이다.

**`DEFAULT CURRENT_TIMESTAMP` — DB가 시각을 채운다**

```python
# 애플리케이션이 시각을 안 넘긴다
conn.execute("INSERT INTO guestbook (name, message) VALUES (?, ?)", (name, message))
#                                    created_at 이 없다!
```

**DB가 알아서 넣는다.** 서버 시계가 여러 대여도 **DB 기준 하나로 통일**된다.

> **⚠ SQLite의 `CURRENT_TIMESTAMP`는 UTC다.** 한국 시각이 아니다.
> 이 실습은 그대로 뒀지만, 실제로는 시간대 처리를 정해야 한다.

**`CREATE INDEX` — 인덱스**

```sql
CREATE INDEX idx_guestbook_created_at ON guestbook (created_at DESC);
```

**왜 필요한가:**

```sql
SELECT * FROM guestbook ORDER BY created_at DESC LIMIT 100;
```

- **인덱스 없으면**: 100만 행을 **전부 읽고 정렬** → 느리다
- **인덱스 있으면**: 이미 정렬된 목록에서 **앞 100개만** → 빠르다

**책의 색인과 같다.** 없으면 처음부터 다 넘겨야 한다.

> **공짜가 아니다.** 인덱스는 **쓰기를 느리게** 하고 **디스크를 더 쓴다**.
> INSERT마다 인덱스도 갱신해야 하니까. **읽기가 많으면 이득, 쓰기만 많으면 손해**다.

#### SQL 4가지만 알면 시작할 수 있다

| 명령 | 하는 일 | 이 실습의 코드 |
|---|---|---|
| **SELECT** | 읽기 | `SELECT id, name, message FROM guestbook ORDER BY ... LIMIT ?` |
| **INSERT** | 쓰기 | `INSERT INTO guestbook (name, message) VALUES (?, ?)` |
| **UPDATE** | 수정 | (이 실습엔 없다) |
| **DELETE** | 삭제 | `DELETE FROM guestbook WHERE id = ?` |

**`SELECT *` 대신 컬럼을 적었다:**

```python
"SELECT id, name, message, created_at FROM guestbook ..."
#        ↑ * 가 아니라 명시
```

**필요한 것만 읽는 게 낫다.** 나중에 큰 컬럼(이미지 등)이 추가돼도 안 딸려온다.

#### `commit()`을 잊으면 안 된다

```python
conn.execute("INSERT INTO guestbook (name, message) VALUES (?, ?)", (name, message))
conn.commit()          # ★ 이게 없으면 저장이 안 된다
```

**가장 흔한 초보 실수다.** INSERT는 됐는데 조회하면 없다 → `commit()`을 안 한 것.

**6단계의 ACID 중 D(지속성)**가 여기서 실현된다. 커밋 전까지는 트랜잭션 안의 임시 상태다.
