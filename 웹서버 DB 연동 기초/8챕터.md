#### NoSQL = "관계형이 아닌 것" 전부

**하나의 기술이 아니라 "나머지 전부"를 묶은 말**이다. 성격이 제각각이다.

| 종류 | 대표 | 저장 형태 | 쓰는 곳 |
|---|---|---|---|
| **문서형** | **MongoDB** | JSON 같은 문서 | 스키마가 자주 바뀌는 데이터 |
| **키-값** | **Redis** | key → value | **캐시**, 세션, 순위표 |
| **컬럼형** | Cassandra | 컬럼 패밀리 | 대용량 로그 |
| **그래프** | Neo4j | 노드·엣지 | 관계망 (친구, 추천) |
| **시계열** | InfluxDB | 시간 → 값 | **센서 데이터**, 모니터링 |

#### MongoDB — 문서형

```javascript
// 관계형이면 컬럼을 미리 정해야 한다
// MongoDB 는 문서마다 구조가 달라도 된다
{ "seq": 1, "name": "이지용", "message": "첫 글" }
{ "seq": 2, "name": "방문자", "message": "안녕", "이미지": "a.jpg" }   // ← 필드 추가 OK!
```

**`03_flask_mongodb`가 이걸 보여준다.**

**SQLite 판과 비교하면:**

| | SQLite (`db.py`) | MongoDB (`db_mongo.py`) |
|---|---|---|
| 연결 | `sqlite3.connect("guestbook.db")` — **파일** | `MongoClient("mongodb://127.0.0.1:27017")` — **서버** |
| 테이블 생성 | `CREATE TABLE ...` **필수** | **없음** (문서 넣으면 생김) |
| 조회 | `SELECT * FROM guestbook` | `col.find({})` |
| 조건 | `WHERE name = ?` | `col.find({"name": "이지용"})` |
| 삽입 | `INSERT INTO ... VALUES (?, ?)` | `col.insert_one({...})` |
| **자동증가 id** | `AUTOINCREMENT` **공짜** | ★ **직접 만들어야 함** |
| 연결 관리 | 요청마다 열고 닫기 | **전역에 하나** 두고 재사용 |

**★ 자동증가 id가 없는 게 실제로 불편하다:**

```python
# db_mongo.py — 직접 만들어야 한다
def _next_seq():
    last = col.find_one(sort=[("seq", DESCENDING)])
    return (last["seq"] + 1) if last else 1
```

MongoDB의 기본 키는 `_id`인데 `ObjectId('507f1f77bcf86cd799439011')` 같은 값이라
**URL에 넣기 길고 사람이 못 읽는다.** 그래서 1,2,3을 손으로 만든다.

> **"관계형이 편한 지점"의 대표 사례**다. NoSQL이 항상 낫지 않다.

#### Redis — 키-값 (성격이 완전히 다르다)

```
SET user:1:name "이지용"
GET user:1:name          → "이지용"
```

**메모리에 저장한다.** 그래서:
- **매우 빠름** (마이크로초)
- **서버 끄면 사라짐** (설정으로 디스크 저장도 가능)

**DB를 대체하는 게 아니라 앞에 둔다:**

```
요청 → Redis 에 있나? ─예→ 즉시 응답 (빠름)
              │
             아니오
              ↓
          PostgreSQL 조회 → Redis 에 저장해 두고 → 응답
```

**이걸 캐시라고 한다.**

#### NoSQL이 항상 좋은 게 아니다

**2010년대 초 "NoSQL이 관계형을 대체한다"는 유행이 있었다. 안 그랬다.**

| | 관계형이 나은 경우 | NoSQL이 나은 경우 |
|---|---|---|
| 데이터 구조 | **정해져 있다** | 자주 바뀐다 |
| 관계 | **여러 표를 조인** | 문서 하나에 다 넣음 |
| 트랜잭션 | **필요** (돈, 재고) | 덜 중요 |
| 규모 | 한 서버로 충분 | **수평 확장** 필요 |

**지금의 결론**: **잘 모르겠으면 PostgreSQL.** 관계형이 기본이고,
특수한 이유가 있을 때 NoSQL을 **추가로** 쓴다.
