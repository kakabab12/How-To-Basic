# 12챕터 — 실습 ③ ★ 파라미터 바인딩 — SQL 인젝션 직접 막아보기

> **웹 서버 + DB 연동 기초** · 12/15챕터 · [📚 목차](README.md)

---

**이 실습에서 가장 중요한 단계다.**

#### 2024년의 코드를 다시 보자

> **[ASP 프로젝트](../개인%20프로젝트/2024년%202학기ASP_DB연동/README.md)의 실제 코드:**
> ```asp
> SQL = "INSERT INTO IdeaTime (Idea) VALUES ('"& X &"')"
> Conn.Execute SQL
> ```
> **문자열을 이어 붙여 SQL을 만든다.** 전 파일이 이 방식이었다.

#### 왜 위험한가

```python
# 절대 이렇게 쓰면 안 된다
name = request.form["name"]
sql = "INSERT INTO guestbook (name, message) VALUES ('" + name + "', '" + message + "')"
```

**공격자가 `name`에 이걸 넣으면:**

```
'); DROP TABLE guestbook; --
```

**만들어지는 SQL:**

```sql
INSERT INTO guestbook (name, message) VALUES (''); DROP TABLE guestbook; --', '메시지')
                                             ─┬─  ───────────┬──────────  ─┬
                                        INSERT 를 끝내고    테이블 삭제!   뒤는 주석
```

**테이블이 통째로 날아간다.**

#### 왜 이런 일이 생기나 — 원인은 하나다

```
사용자 입력이 "데이터"가 아니라 "SQL 문법"으로 해석된다
```

`'`(따옴표)가 **문자열을 끝내는 문법**이라서, 사용자가 `'`를 넣는 순간
**SQL의 구조를 바꿀 수 있게 된다.**

#### 해결 — 파라미터 바인딩

```python
# db.py 의 실제 코드
conn.execute(
    "INSERT INTO guestbook (name, message) VALUES (?, ?)",   # ← 값 자리에 ?
    (name, message),                                          # ← 값은 따로!
)
```

**`?`를 쓰면:**

```
SQL 문장   → DB 에 먼저 보낸다 → DB 가 "구조"를 확정한다
값        → 나중에 따로 보낸다 → DB 는 이걸 "데이터"로만 취급

→ 값이 무슨 문자를 담고 있든 절대 문법이 될 수 없다
```

**따옴표를 이스케이프하는 게 아니다.** 값과 문법을 **애초에 다른 통로로** 보낸다.
**원리적으로 안전**하다.

#### 실제로 공격해서 확인했다

```bash
curl -X POST http://localhost:5000/api/entries \
  -H "Content-Type: application/json" \
  -d '{"name":"해커'"'"'); DROP TABLE guestbook; --","message":"테이블 날려보자"}'
```

**결과:**

```json
{ "id": 5,
  "name": "해커'); DROP TABLE guestbook; --",     ← 그냥 이름으로 저장됨
  "message": "테이블 날려보자" }
HTTP 201
```

**공격 후 테이블 확인:**

```bash
curl http://localhost:5000/health
→ {"entries": 5, "status": "ok"}      ← 테이블 멀쩡, 오히려 글이 하나 늘었다
```

> **공격 문자열이 "이름"으로 저장됐을 뿐이다.** 이게 파라미터 바인딩이 작동한다는 증거다.

#### 다른 언어도 전부 같은 원리

```python
# Python + SQLite
cur.execute("SELECT * FROM t WHERE id = ?", (user_id,))
# Python + PostgreSQL
cur.execute("SELECT * FROM t WHERE id = %s", (user_id,))
```
```javascript
// Node.js + MySQL
conn.query("SELECT * FROM t WHERE id = ?", [userId]);
```
```asp
' 클래식 ASP 도 방법이 있었다!
Set Cmd = Server.CreateObject("ADODB.Command")
Cmd.CommandText = "DELETE FROM tblCategories WHERE CategoryID = ?"
Cmd.Parameters.Append Cmd.CreateParameter("cid", 200, 1, 50, X1)
```

> **2024년 ASP 실습에서도 할 수 있었다.** 교재가 안 가르쳤을 뿐이다.

#### 보너스 — XSS도 확인했다

**SQL 인젝션의 사촌**이다. 이번엔 **HTML**에 코드를 심는다.

```bash
curl -X POST localhost:5000/api/entries \
  -d '{"name":"<script>alert(1)</script>","message":"..."}'
```

**HTML을 확인해 보니:**

```html
&lt;script&gt;alert(1)&lt;/script&gt;      ← 이스케이프됨. 글자로 보일 뿐 실행 안 됨
```

**Jinja2가 자동으로 이스케이프**해 준다. `<`를 `&lt;`로 바꿔서 **문법이 아니라 글자**로 만든다.

> **SQL 인젝션과 XSS는 완전히 같은 문제**다:
> **"데이터가 코드로 해석되는 것"**. 대응도 같다 — **데이터와 코드를 분리**한다.
> SQL은 파라미터 바인딩, HTML은 이스케이프.

#### MongoDB는 어떤가

```python
# db_mongo.py
col.insert_one({"name": name, "message": message})
```

**SQL 인젝션이 원리적으로 없다.** 쿼리가 **문자열이 아니라 딕셔너리(BSON)**라서
값이 문법이 될 수 없다.

> **단, NoSQL 인젝션이 아예 없는 건 아니다.** 사용자 입력을 쿼리 **구조**에 그대로 넣으면
> `{"$ne": null}` 같은 연산자를 주입당할 수 있다. **값 자리에 넣는 이 코드는 안전**하다.
