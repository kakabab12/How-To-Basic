# 2챕터 — Plan A → Plan B — 파일 계보 읽는 법

> **젯슨 나노 객체인식 분류 + 르로봇** · 2/15챕터 · [📚 목차](README.md)

---

`jetson_main1(2).py`, `jetson_main1(5).py`... 이런 파일이 왜 이렇게 많은가?
**하드웨어가 고장 나서 설계를 두 번 바꿨기 때문**이다. 본인이 쓴 `코드설명.md`가 증언한다:

> 초기 CSI 카메라 기반 설계에서 발생한 하드웨어 충돌을 극복하고, 안정적인 USB 듀얼 카메라
> 아키텍처와 TensorRT AI 가속을 이뤄낸 결과물입니다.

> 고장 난 CSI 포트를 버리고 Intel RealSense와 일반 USB 웹캠(다이소)을 동시에 구동하도록
> 설계되었습니다.

#### 계보

| 세대 | 파일 | 정체 |
|---|---|---|
| **0세대** | `jetson_main1.py` | 초기 프로토타입. 아두이노 시리얼 규격(`MOVE:RIGHT`) 확립, TensorRT 최초 로드 테스트 |
| **1세대 (Plan A)** | `jetson_csi_final.py`, `jetson_main1(9).py` | **CSI 카메라(IMX219) + GStreamer** (`nvarguscamerasrc`) 하드웨어 인코딩 지향 |
| **중간 시도** | `RIP/jetson_main1(2)~(8).py`, `jetson_main1-2.py`, `jetson_mainsppd.py` | 폐기된 것들 (**`RIP` 폴더 = Rest In Peace**) |
| **2세대 (Plan B)** | **`jetson_USB.py`, `jetson_USB2.py`** | **최종 완성본.** RealSense + USB 웹캠. 전시회에서 실제 사용 |

본인 설명이 Plan A를 이렇게 평가한다:

> nvarguscamerasrc 등 GStreamer 파이프라인을 활용해 CSI 카메라의 하드웨어 인코딩 영상을
> 가져오도록 설계된 고성능 지향 코드였습니다. **(보고서의 '설계 변경 과정'을 증명하는
> 소중한 자료입니다.)**

> **읽는 순서 권장**: `jetson_USB2.py`(최종본) 하나만 정독하면 된다. 나머지는 **왜 그렇게
> 됐는지**가 궁금할 때 비교용으로 본다. 특히 6단계에서 계보 비교가 버그를 잡아낸다.

#### 진단 도구들 — 하드웨어와 싸운 흔적

| 파일 | 역할 |
|---|---|
| `cam_check.py` | 듀얼 카메라 멀티스레딩 추론이 **락 없이 도는지** 사전 점검 |
| `imx.py` | `cv2.VideoCapture`만 여는 **초경량** 마운트 확인 |
| `testgpu.py` | GPU 사용 가능 확인 |
| `test_dofBot.py` | 로봇팔 개별 점검 |
| `camera_test.jpg` | 캡처 결과물 |

**메인 서버를 켜기 전에 문제를 좁히는 도구를 따로 만들어 둔 것**이 좋은 습관이다.
듀얼 카메라 + GPU + 시리얼이 한꺼번에 도는 코드에서 문제가 나면 원인을 특정할 수 없다.

#### 실제 운영 기록

`smart_factory.log`, `factory_gate.log`, `factory_operations.log` — **실제로 돌린 로그**가
남아 있다. 코드가 진짜 동작했다는 증거다.
