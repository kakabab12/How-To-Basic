**카메라당 스레드 2개**(캡처 + 추론) × 2대 = **4개**. 여기에 FastAPI가 메인.

```
capture1_loop  ──▶ frame1 (lock) ──▶ inference1_loop ──▶ latest1_frame (lock) ──▶ /video_feed/1
capture2_loop  ──▶ frame2 (lock) ──▶ inference2_loop ──▶ latest2_frame (lock) ──▶ /video_feed/2
```

#### 왜 캡처와 추론을 분리했나

라즈베리파이 편(`sucess.py`)은 **한 스레드**에서 `get_frame()` → 추론을 순서대로 했다.
젯슨은 **나눴다.** 이유:

- **카메라는 30fps로 프레임을 밀어낸다.** 추론이 10fps면 20fps어치가 버퍼에 쌓여 지연이 생긴다.
- 캡처 스레드가 **최신 프레임만 덮어쓰면** 추론은 항상 "지금"을 본다.
- 두 카메라의 추론 속도가 달라도 **서로를 기다리지 않는다.**

```python
def inference2_loop():
    while True:
        with frame2_lock:
            frame = latest2_raw.copy() if latest2_raw is not None else None
        if frame is None:
            time.sleep(0.01); continue
        try:
            annotated = process_frame(frame, model2, cam_id=2)
            with frame2_lock:
                latest2_frame = annotated
        except Exception as e:
            print(f"[ERROR] CAM2(USB Webcam) 에러: {e}")
        time.sleep(0.01)
```

**`try/except`로 감싼 게 중요하다.** 카메라 하나가 에러를 내도 **다른 카메라와 서버는
계속 돈다.** 전시회에서 한쪽이 죽어도 시연이 이어지게 하는 방어다.

`cam_check.py`가 존재하는 이유도 여기 있다 — 이 4스레드 구조가 **락 없이(deadlock 없이)
도는지** 미리 검증하는 도구다. 본인 설명:

> 메인 서버를 켜기 전에 리얼센스와 웹캠이 충돌 없이 YOLO 추론 루프를 잘 도는지,
> 프레임 락(Lock)이 걸리지 않는지 점검합니다.
